Background
Recent commits added Kitty keyboard protocol flag 2 support:
a2f032a4
- Added flag 2 to Kitty protocol (\x1b[>3u
), which enables key release event reporting41fe329c
- Attempted to fix double-processing by filtering release events
The fix in 41fe329c
used this approach:
if (isKeyRelease(data) && !this.focusedComponent.wantsKeyRelease) {
return; // Drop the input
}
Where isKeyRelease()
checks if the data string contains any release event pattern:
export function isKeyRelease(data: string): boolean {
if (data.includes(":3u") || data.includes(":3~") || ...) {
return true;
}
return false;
}
Root Cause
Over SSH (and sometimes locally with fast typing), the terminal batches multiple events into a single data chunk:
"\x1b[97u\x1b[97;1:3u\x1b[98u\x1b[98;1:3u"
^press a ^release a ^press b ^release b
When this batched input arrives, isKeyRelease()
returns true
because the string contains :3u
. The code then drops the entire batch, including the valid key press events.
This explains why:
- The issue is worse over SSH (more batching due to network latency)
- Fast typing is affected more (more events batched together)
- It is a recent regression (started with Kitty flag 2 addition)
Solution
Replace the boolean isKeyRelease()
check with a new filterKeyReleases()
function that uses regex to remove only the release event sequences while preserving press events:
export function filterKeyReleases(data: string): string {
return data
.replace(/\x1b\[\d+(?:;\d+)?:3u/g, "") // CSI u releases
.replace(/\x1b\[\d+(?:;\d+)?:3~/g, "") // Functional key releases
.replace(/\x1b\[1;\d+:3[ABCDHF]/g, ""); // Arrow/Home/End releases
}
Updated input handling:
// Before: drops entire batch if ANY release event present
if (isKeyRelease(data) && !wantsKeyRelease) return;
// After: removes only release sequences, keeps presses
const filtered = wantsKeyRelease ? data : filterKeyReleases(data);
if (filtered.length === 0) return;

The isKeyRelease() check used includes() which returned true if ANY event in a batch was a release, causing the entire batch to be dropped. Fix: Add filterKeyReleases() that uses regex to remove only release sequences while preserving press events in batched input.
